Basics · Part 1

What is Redis?

Redis is a place to store data — like a database — but with one big difference: it keeps everything in memory (RAM), not on a disk.

Why does that matter? Think of it like this. A disk is a big cupboard in another room: lots of space, but you have to walk there every time. Memory is the top of your own desk: small, but everything on it is within reach of your hand. Redis keeps your data on the desk.

Because of this, Redis can answer most questions in less than one millisecond. For comparison, a normal database usually takes 10–100 milliseconds. That difference — felt millions of times a day — is why almost every big app you use has Redis somewhere inside it.

The one sentence to remember: Redis = your data on the desk, not in the cupboard. Fast because it never walks to the other room.
Basics · Part 2

Why is Redis so fast?

Three simple reasons:

1 · Memory instead of disk

Reading from memory is roughly 10,000× faster than reading from a disk. This one fact is most of the speed. Redis never waits for a disk to answer a question.

2 · One very fast cashier

Redis handles commands one at a time, like a single cashier at a counter. That sounds slow — but this cashier serves each customer in a microsecond, and because there is only one, nobody argues over the cash box. No locks, no fights, no waiting on each other.

3 · Simple questions, direct answers

Redis doesn't understand complicated queries and doesn't need to plan anything. You say "give me the value of this key" — it jumps straight to it. The commands map almost directly to memory operations.

Basics · Part 3

Why can Redis handle millions of requests?

Follow the math of the single cashier. If serving one customer takes 1 microsecond, one cashier can serve up to a million customers per second — not because the cashier multitasks, but because each customer takes almost no time at all.

Real numbers: one ordinary Redis server comfortably handles 100,000+ commands per second. And when even that is not enough, Redis Enterprise Cloud can split your data across several servers (each with its own cashier) — 6 servers means roughly 6× the desk space and 6× the speed. Your code doesn't change; the splitting happens underneath.

1Each command is tiny (microseconds)
2So one server handles 100K+ per second
3Need more? Add servers — the data splits across them
4Millions of requests per second, no code changes
Basics · Part 4

How does Redis work internally?

Here is the whole architecture, in plain words. Picture a post office with one clerk:

1Your app connects — it opens one line to Redis and keeps it open (like keeping a phone call running instead of redialing).
2Commands wait in one line — every request from every app joins a single queue.
3The clerk serves one at a time — this is the famous "single thread." One at a time, but each takes a microsecond.
4Data sits in labeled boxes — memory is organized like shelves of labeled boxes: the label is the key ("user:42"), the content is the value.

Two more pieces complete the picture:

Saving to disk (so a restart doesn't lose everything): Redis can take a photo of all its boxes every few minutes, and it can also keep a diary of every change. After a crash, it reloads the photo and replays the diary. In Redis Enterprise Cloud, this is a setting you switch on — no setup work.

Copies (so one dead machine doesn't stop you): Redis can run a live copy on a second machine that mirrors everything within milliseconds. If the main machine dies, the copy takes over in seconds — automatically. In Redis Enterprise Cloud, this is a checkbox.

Simple example, start to finish: your app asks GET user:42. The command travels down the open line, waits its (microsecond) turn in the queue, the clerk reads box "user:42" from the shelf, and the answer travels back. Total time: well under a millisecond.
Basics · Part 5

Why do big companies use Redis?

Because every one of them has the same two problems: the same data being asked for millions of times, and users who expect everything instantly. Here is where Redis sits inside apps you use every day:

CompanyWhere Redis helpsIn simple words
InstagramProfiles & feedsA celebrity's profile is viewed millions of times a day. Redis keeps a copy in memory, so the database answers once, not a million times.
YouTube-style appsView countersA viral video gets thousands of views per second. Counting in memory (+1, +1, +1...) is safe and instant; the total is saved to the database calmly, later.

Feature 1

Redis as a Cache

1 · The real problem — Instagram

A cricket star with 50 million followers posts a photo. In the next hour, 2 million people open his profile. His profile data (name, photo, bio, follower count) is EXACTLY the same for all 2 million viewers — yet without a cache, the main database runs the same lookup 2 million times. Pages get slow, and the database struggles under pointless repeated work.

2 · How Redis solves it

Imagine you visit the same website every day. Instead of asking the database for the same information every time, Redis keeps a copy in memory. Memory is much faster than a database, so the page appears almost instantly — and the database only works when something actually changes.

The recipe is called cache-aside, and it is three steps: (1) check Redis first; (2) if the copy is there, return it — done in under a millisecond; (3) if not, ask the database once, save the answer in Redis with an expiry time, and return it. The next 1,999,999 viewers hit step 2.

3 · What Redis is doing behind the scenes

Redis stores the profile as one labeled box: key user:virat, value = the profile as JSON text. The expiry time (called a TTL — "time to live") is a countdown attached to the box: when it reaches zero, Redis throws the box away by itself. This keeps copies fresh and memory clean, with zero cleanup code.

SET user:virat '{"name":"Virat","followers":50000000,...}' EX 600   # keep a copy for 10 minutes
GET user:virat                                                       # 2 million fast reads
DEL user:virat                                                       # profile edited? delete the copy — next read is fresh

4 · Spring Boot implementation: in the Spring Boot tab, this whole feature becomes two annotations — @Cacheable (check Redis first, fill it on a miss) and @CacheEvict (delete the copy when the profile changes).

5 · Best practices
  • Always set an expiry (TTL). A copy without an expiry becomes stale forever and leaks memory.
  • When the original changes, DELETE the copy (don't try to update it) — the next read refills it correctly.
  • Cache data that is read often and changes rarely: profiles, product pages, settings.
  • Never let Redis be the only home of important data — the database stays the source of truth.
Feature 2

Redis Pub/Sub

1 · The real problem — a payment app

Your friend sends you ₹500 on a payment app. You are staring at the app — and you expect the "₹500 received!" message to pop up immediately, not whenever the app next checks the server. The naive fix — the app asking "anything new?" every few seconds — is late by design, drains your battery, and floods the servers with millions of empty "anything new?" calls.

2 · How Redis solves it

Pub/Sub turns the direction around: instead of your app repeatedly ASKING, the server TELLS your app the moment something happens. Your app subscribes to a channel with your name on it (like tuning a radio to your personal station). When money arrives, the payment server publishes one small message to that channel — and your app hears it within milliseconds. The popup appears while your friend is still holding their phone.

3 · What Redis is doing behind the scenes

Redis keeps a simple list of "who is currently listening to which channel." When a message is published, Redis instantly forwards it to every current listener — and then forgets it. Pub/Sub is live radio: nothing is recorded. If your phone was off, that particular shout is gone (which is fine — the transaction itself is safely in the database; only the live popup was missed).

SUBSCRIBE payments:user:ramesh                       # the app listens on its channel
PUBLISH payments:user:ramesh '{"received":500,"from":"suresh"}'   # server shouts — all listeners hear it instantly

4 · Spring Boot implementation: in the Spring Boot tab — a publisher service (one line: convertAndSend) and a listener container that receives messages and pushes them to the user's phone.

5 · Best practices
  • Use Pub/Sub for "right now" moments: popups, live counters, "user is typing…".
  • Never use it as the ONLY record of something important — it forgets. Store the fact in a database first, shout second.
  • One channel per user (payments:user:ramesh) keeps routing simple.
Feature 3

Redis Search

1 · The real problem — Amazon

You type "iph" into the Amazon search box — and before you finish the word, "iPhone 15", "iPhone charger", "iPhone case" appear as suggestions. That means the search runs on every keystroke, for millions of shoppers at once, and each answer must come back in a few milliseconds. Running text-matching queries on the main product database for every keystroke would crush it — and the suggestions would lag behind your typing.

2 · How Redis solves it

Redis Enterprise Cloud includes a real search engine. You tell it once: "index my products — the name is searchable text, the price is a number, the category is a label." From then on, every product you store is automatically searchable. Typing "iph" becomes one command that answers from memory in ~2 milliseconds — fast enough to search on literally every keystroke, sorted so bestsellers come first.

3 · What Redis is doing behind the scenes

When you save a product, Redis quietly updates an index — think of the index at the back of a textbook: instead of reading the whole book to find "iPhone", you look up "iPhone" in the index and jump straight to the right pages. Redis maintains that back-of-the-book index automatically for every product, in memory. Searching never scans your data; it jumps.

FT.CREATE idx:products ON HASH PREFIX 1 "product:" SCHEMA
    name TEXT SORTABLE  price NUMERIC SORTABLE  category TAG      # define the index ONCE

FT.SEARCH idx:products "iph*" SORTBY sales DESC LIMIT 0 5          # every keystroke — ~2ms
FT.SEARCH idx:products "@category:{mobiles} @price:[10000 30000]"  # filters work too

4 · Spring Boot implementation: in the Spring Boot tab — creating the index at startup and a /search REST endpoint that calls FT.SEARCH with what the user typed.

5 · Best practices
  • The search index is a fast COPY — the main database stays the source of truth. You can always rebuild the index.
  • On the app side, wait ~150ms of typing pause before searching (called debouncing) — fewer, better queries.
  • Sort results by something users care about (sales, popularity) — not just text match.
Feature 4

Redis Geospatial

1 · The real problem — Amazon delivery

You order a package, and the app shows: "Your delivery partner is 1.2 km away." Behind that little line is a hard question: out of thousands of moving delivery partners in your city, which ones are near this exact address, right now? Calculating your distance to every single partner, one by one, for every customer who opens the tracking screen, is far too slow — and partner locations change every few seconds.

2 · How Redis solves it

Redis has location built in. Each delivery partner's app sends its position every few seconds, and Redis stores it (updating in place — no delete-and-re-add). Then "who is within 3 km of this address?" is one command that answers in about a millisecond, already sorted nearest-first, with distances included. The tracking screen just asks and displays.

3 · What Redis is doing behind the scenes

Redis converts each latitude+longitude into one clever number, built so that places near each other get numbers near each other. All partners then live in one sorted collection ordered by that number. "Near me" stops being "measure everyone's distance" and becomes "look at my number's neighbors" — a tiny, fast lookup instead of a city-wide calculation.

GEOADD partners:pune 73.8567 18.5204 "partner:raju"     # update position (overwrite in place)
GEOADD partners:pune 73.8601 18.5310 "partner:salim"

GEOSEARCH partners:pune FROMLONLAT 73.8560 18.5195
          BYRADIUS 3 km ASC WITHDIST                     # nearest partners within 3 km, sorted

4 · Spring Boot implementation: in the Spring Boot tab — a location-update endpoint (PUT) and a "partners near me" endpoint (GET) built on Spring's geo operations.

5 · Best practices
  • Location is personal data — store it only when needed, and remove entries that stop updating (a partner who went offline).
  • Keep one geo key per city or zone (partners:pune) — searches stay small and fast.
  • Update positions by simply calling GEOADD again — it overwrites in place.
Feature 5

Redis Data Structures

Redis is not just "key → text." A key can hold a shape of data, and each shape has its own smart commands. Picking the right shape is the real skill. Here are the five that matter — each introduced by the app that needs it:

String — one box, one value (YouTube)

1 · The problem

A video goes viral: 8,000 views per second. Doing "read the count, add 1, write it back" from many servers at once loses counts — two servers read "1000" at the same moment, both write "1001", and one view vanishes.

2 · How Redis solves it

The INCR command adds 1 atomically — meaning the read-add-write happens as one unbreakable step inside Redis. 8,000 INCRs per second from 20 servers, and not one view is lost. (A String can also hold text or JSON — it's the same shape the cache in Feature 1 uses.)

3 · Behind the scenes

Remember the single cashier: commands run one at a time, so two INCRs can never overlap. Atomic safety isn't extra machinery — it falls out of the architecture.

INCR video:abc123:views        # +1, safe under any traffic — returns the new count
GET  video:abc123:views

Hash — one object, many fields (payment app)

1 · The problem

A user's wallet has a balance, a currency, a status, and a last-updated time. Stored as one JSON text, changing JUST the balance means reading the whole thing, editing it, writing the whole thing back — wasteful and racy.

2 · How Redis solves it

A Hash is a box with labeled compartments — one key, many field→value pairs. You can read or update ONE compartment without touching the rest, and HINCRBY changes a number field atomically.

3 · Behind the scenes

Redis stores the fields in a small map inside the value. Field operations jump straight to the one compartment — no read-modify-write of the whole object.

HSET wallet:ramesh balance 2500 currency INR status ACTIVE
HINCRBY wallet:ramesh balance -500        # sent ₹500 — ONE field changed, atomically
HGET wallet:ramesh balance

List — an ordered line (Facebook)

1 · The problem

Your Facebook notifications: newest on top, load 20 at a time, scroll for more. Rebuilding that ordered list from the database on every open is repeated, pointless work.

2 · How Redis solves it

A List is an ordered row of items you push onto and read in pages. New notification → push to the front. Open the screen → read items 0–19. Scroll → read 20–39. Trim the tail so it never grows forever.

3 · Behind the scenes

The list lives in memory as a chain — pushing to the front and reading a range from the front are both instant, no matter how long the history.

LPUSH notifications:ramesh '{"type":"comment","from":"suresh"}'   # newest at the front
LRANGE notifications:ramesh 0 19                                   # first page
LTRIM notifications:ramesh 0 499                                   # keep only the newest 500

Set — a bag of unique items (Instagram)

1 · The problem

Users double-tap to like — and sometimes double-tap twice, fast, on a laggy network. Counted naively, one person becomes two likes. And the app must know instantly whether to show YOUR heart as red on each post.

2 · How Redis solves it

A Set holds unique items — adding the same item twice does nothing, by definition. Add the user to the post's likers set: the command itself answers "1 = new like" or "0 = already liked." The like count is just the set's size; "is my heart red?" is one membership check.

3 · Behind the scenes

The duplicate check and the write are ONE atomic step — there is no gap where a second tap can sneak in. That's why the double-count bug is impossible here, not just unlikely.

SADD post:9001:likers ramesh        # returns 1 (new like) — second tap returns 0
SCARD post:9001:likers               # like count
SISMEMBER post:9001:likers priya     # should Priya's heart be red? 1/0

Sorted Set — unique items with scores, always in order (stock trading)

1 · The problem

A trading app's "Top Gainers" screen: stocks ranked by today's % change, updating every second as prices move. Re-sorting thousands of stocks in the database every second, for every user watching, is hopeless.

2 · How Redis solves it

A Sorted Set stores each member with a number (its score) and keeps everything sorted at all times. Update a stock's score and it slides to its new position instantly. "Top 10 gainers" is a read of the first 10 — never a sort. A stock's exact rank is a single lookup.

3 · Behind the scenes

Redis keeps the members in a structure that maintains order as things change (like a self-organizing bookshelf). Sorting is never "done" — it is simply always true.

ZADD gainers 4.2 "TCS"  2.8 "INFY"  -1.3 "WIPRO"    # score = today's % change
ZADD gainers 5.1 "TCS"                                # TCS jumps — position updates itself
ZREVRANGE gainers 0 9 WITHSCORES                      # top 10 gainers, right now
ZREVRANK gainers "INFY"                               # INFY's exact rank
ShapeMemory hookExample above
StringOne value; numbers count atomicallyYouTube view counter
HashOne object, labeled compartmentsPayment wallet
ListOrdered line; push front, read pagesFacebook notifications
SetUnique items; duplicates impossibleInstagram likes
Sorted SetScored items, always sortedStock top gainers
5 · Best practices (for all five)
  • Pick the shape that matches the question, not the data: "how many unique…?" = Set; "top N by…?" = Sorted Set; "newest first, in pages" = List.
  • Name keys like folders: post:9001:likers — type, ID, purpose.
  • Cap growing collections (LTRIM lists; expire old sorted sets) — no key should grow forever.
Feature 6 · The most important pattern in this guide

INCR — Atomic Counters, and how Redis stays safe under a storm

This one deserves its own section, because it hides a bug that every backend engineer meets sooner or later — and Redis is one of the cleanest ways to make the bug impossible.

1 · The real problem — a flash sale on Flipkart

Flipkart puts up a flash sale: only 100 phones at ₹9,999. At 12:00:00 sharp, 50,000 people tap "Buy" in the same second. The rule is simple: sell exactly 100, then stop. Sell 101, and the company owes an angry customer a phone it doesn't have. Sell 99, and the marketing team is furious.

This sounds easy — just count down from 100. It is not easy. Let's see why.

2 · Why simple counting breaks — the race condition

Single-threaded world (safe, but only in theory): if only ONE process, ONE thread, ONE server ever touches the counter, then "read stock, subtract 1, write stock" is safe — nobody else is looking at it during those three steps.

Multi-threaded / multi-server world (what real backends actually are): Flipkart runs its checkout on 50 servers behind a load balancer, and each server handles many requests at once. Now two servers can read the SAME value at the SAME moment. Watch:

TimeServer A (Ramesh)Server B (Suresh)Stock in DB
12:00:00.001reads stock → sees 100—100
12:00:00.002—reads stock → sees 100 too!100
12:00:00.003writes stock = 99, sells to Ramesh—99
12:00:00.004—writes stock = 99, sells to Suresh99 ❌

Two phones were sold, but the counter only dropped by one. Do this 50,000 times per second and Flipkart sells 130 phones out of 100 — and finds out only when the warehouse ships. This is called a race condition: two threads racing each other, both reading old data because the read and the write are separate steps.

3 · How Redis solves it — INCR / DECR is ONE step

Remember the single cashier from Section 4? Every Redis command is served one at a time. So when you say DECR stock:flashsale, Redis does "read the number, subtract 1, write it back" as ONE unbreakable step — no other command can slip in between. The race condition is not "unlikely" here. It is impossible by design.

Even better: DECR returns the NEW value. So the checkout code becomes: "call DECR, look at the number I got back — if it's ≥ 0 the sale succeeds, if it's < 0 the sale fails (sold out) and I add 1 back." No reads, no gaps, no double sells.

SET stock:flashsale 100                    # start of the sale
DECR stock:flashsale                        # customer 1 → returns 99  ✓ sell
DECR stock:flashsale                        # customer 2 → returns 98  ✓ sell
# ... 98 more customers ...
DECR stock:flashsale                        # customer 100 → returns 0   ✓ sell (last one)
DECR stock:flashsale                        # customer 101 → returns -1  ✗ sold out — refund the decrement
INCR stock:flashsale                        #            → back to 0, ready to reject the rest

50,000 clicks per second, 50 servers, one Redis counter — and exactly 100 phones are sold. Every time.

4 · So — where does this pattern appear in real apps?

Once you see it, you see it everywhere:

WhereWhat's being counted atomically
Flash sale / limited stockUnits remaining — DECR until it hits 0
YouTube video viewsINCR the view counter — thousands per second, none lost
Instagram likes countINCR when a viral post gets liked 800×/second
Ticket booking (BookMyShow)DECR seats-available for a movie showing
Payment app rate limitsINCR attempts; block after 5/minute
API quotasINCR a per-user counter; reject past the daily limit
Payment idempotency counterEnsure the same request isn't processed twice

Every one of these is the same shape: many servers, one shared number that must move up or down safely. Redis's atomic commands (INCR, DECR, INCRBY, DECRBY, and their hash cousins HINCRBY, sorted-set cousins ZINCRBY) are the tool.

5 · Spring Boot implementation: in the Spring Boot tab, chapter 7 builds exactly this flash-sale feature — with the safe check-and-refund flow.

Best practices
  • Never do "read the counter in Java, then write it back" for shared numbers. That's the race condition. Use INCR / DECR — one command, atomic.
  • When over-decrementing is possible (stock, quotas), check the return value and INCR back if the sale fails. One extra command is nothing.
  • Java's AtomicInteger is only safe inside ONE server. Across many servers, only a shared atomic counter (Redis) works.
  • Give counters that represent temporary events (flash sales, rate-limit windows) a TTL so they clean themselves up.

1

Connect Spring Boot to Redis Enterprise Cloud

📖 Do this once — every feature below uses it

When you create a database in Redis Enterprise Cloud, the console shows you three things: a host (a long address ending in redis-cloud.com), a port (a number), and a password. That's all we need.

The steps
  1. Add one dependency — it brings the Redis driver and all of Spring's Redis helpers.
  2. Put host, port, and password in application.yml — the password comes from an environment variable so it never enters your code or Git.
  3. Create one RedisTemplate bean — the object your services use to talk to Redis. Built once, reused everywhere.
// pom.xml — Spring Boot 4 on Java 25
<properties>
    <java.version>25</java.version>
    <spring-boot.version>4.0.0</spring-boot.version>
</properties>

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-redis</artifactId>
</dependency>
# application.yml — values from the Redis Enterprise Cloud console
spring:
  data:
    redis:
      host: ${REDIS_HOST}            # e.g. redis-12345.c256.ap-south-1.ec2.redns.redis-cloud.com
      port: ${REDIS_PORT:12345}
      password: ${REDIS_PASSWORD}     # set on the server as an environment variable — NEVER in code
      ssl:
        enabled: true                 # the connection is encrypted
      timeout: 2s                     # if Redis doesn't answer in 2s, fail loudly instead of hanging
// RedisConfig.java — one shared template for the whole app
@Configuration
public class RedisConfig {

    @Bean
    public RedisTemplate<String, Object> redisTemplate(RedisConnectionFactory factory) {
        RedisTemplate<String, Object> template = new RedisTemplate<>();
        template.setConnectionFactory(factory);
        template.setKeySerializer(new StringRedisSerializer());                 // keys stay human-readable
        template.setValueSerializer(new GenericJackson2JsonRedisSerializer());   // Java objects ↔ JSON
        template.setHashKeySerializer(new StringRedisSerializer());
        template.setHashValueSerializer(new GenericJackson2JsonRedisSerializer());
        return template;
    }
}
What is happening underneath
  1. Spring reads the yml, opens a small pool of connections to your cloud database, and keeps them open — like keeping a phone line running instead of redialing for every sentence. Every command reuses these lines.
  2. The serializers decide how Java objects become bytes. With these settings, keys look like user:virat and values look like readable JSON in any Redis browser — which makes debugging painless.
  3. Test it: run the app, then in the Redis Cloud console's built-in CLI type PING — it answers PONG. You are connected.
Best practices
  • The password lives ONLY in environment variables. If it ever lands in Git, rotate it immediately — bots scan public repositories for exactly this.
  • Keep the 2s timeout — a hanging cache must fail fast, not slow your whole app quietly.
  • One template bean for the whole app. Never create connections per request.
2

Redis as Cache — the Instagram profile

📖 Feature 1 from the concepts tab
1 · The problem

The cricket star posts; 2 million people open his profile in an hour. Without a cache, our database runs the same profile lookup 2 million times — same question, same answer, full price every time. Profile pages take ~100ms and the database burns CPU on pure repetition.

2 · How Redis solves it

Keep a copy of the profile in Redis for 10 minutes. The first viewer pays the database price once; the next 1,999,999 viewers get the memory copy in under a millisecond. When the star edits his bio, we delete the copy — the very next viewer refills it fresh. In Spring Boot, this entire logic is two annotations.

📖 The story behind the code

CacheConfig is the "rulebook" — we tell Spring: "any time someone asks you to cache something, use Redis, keep it for 10 minutes, and store it as JSON." Written once, obeyed everywhere.

UserService is the story: @Cacheable tells Spring "check the fridge (Redis) before you cook (database). If the dish is there, serve it. If not, cook it, put a copy in the fridge on your way out." @CacheEvict is "the recipe changed — throw out the old dish so the next order gets the new one." That's it. No cache code in your business logic at all.

// CacheConfig.java — turn caching on, point it at Redis, set a 10-minute expiry
@Configuration
@EnableCaching
public class CacheConfig {

    @Bean
    public RedisCacheManager cacheManager(RedisConnectionFactory factory) {
        RedisCacheConfiguration config = RedisCacheConfiguration.defaultCacheConfig()
            .entryTtl(Duration.ofMinutes(10))
            .serializeValuesWith(SerializationPair.fromSerializer(
                new GenericJackson2JsonRedisSerializer()));
        return RedisCacheManager.builder(factory).cacheDefaults(config).build();
    }
}
// UserService.java — the two annotations ARE the feature
@Service
public class UserService {
    private final UserRepository userRepository;   // the main database

    @Cacheable(value = "users", key = "#handle")     // check Redis first; fill it on a miss
    public UserProfile getProfile(String handle) {
        return userRepository.findByHandle(handle)     // runs ONLY when Redis doesn't have it
            .orElseThrow(() -> new UserNotFoundException(handle));
    }

    @CacheEvict(value = "users", key = "#handle")   // profile changed → delete the copy
    public UserProfile updateBio(String handle, String newBio) {
        return userRepository.updateBio(handle, newBio);
    }
}
// UserController.java — the REST API
@RestController
@RequestMapping("/api/v1/users")
public class UserController {
    private final UserService userService;

    @GetMapping("/{handle}")                                    // GET /api/v1/users/virat
    public UserProfile getProfile(@PathVariable String handle) {
        return userService.getProfile(handle);
    }

    @PatchMapping("/{handle}/bio")
    public UserProfile updateBio(@PathVariable String handle, @RequestBody BioRequest request) {
        return userService.updateBio(handle, request.bio());
    }
}
What is happening underneath
  1. @Cacheable makes Spring build the key users::virat and check Redis BEFORE running your method. Copy found → the method body never runs. Not found → method runs, and Spring saves the result to Redis on the way out. You wrote zero cache code.
  2. @CacheEvict deletes users::virat the moment the bio changes — so nobody sees a stale bio for more than an instant.
  3. The 10-minute TTL is the safety net: even if an eviction is ever missed, the copy heals itself within 10 minutes.
Best practices
  • Delete-on-change (@CacheEvict) beats updating the cache in place — deleting can never write a wrong value.
  • Keep the TTL even with perfect eviction. It forgives every future bug.
  • Watch your hit rate (hits ÷ total reads). Below ~80% means you're caching the wrong things or expiring too fast.

Redis Pub/Sub — the “₹500 received!” popup

📖 Feature 2 from the concepts tab
1 · The problem

Suresh sends Ramesh ₹500. Ramesh is looking at his payment app and expects the popup now. But our backend runs on 20 servers — the server that processed Suresh's payment is NOT the server holding Ramesh's live connection. How does server 9 tell server 3 "push a popup to Ramesh"?

2 · How Redis solves it

Redis becomes the shared loudspeaker between all 20 servers. The server that processed the payment PUBLISHES one small message to the channel payments:user:ramesh. Every server SUBSCRIBES to payment channels; the one actually holding Ramesh's connection hears the message and pushes the popup. Milliseconds, end to end.

📖 The story behind the code

Think of a school with 20 classrooms and one central loudspeaker (Redis). A teacher in classroom 9 needs to send a message to a student named "Ramesh" — but doesn't know which classroom Ramesh is sitting in. So the teacher speaks into the loudspeaker: "Message for Ramesh!" Every classroom hears it. The one classroom where Ramesh actually is delivers the message; the other 19 shrug and move on.

PaymentNotifier is the teacher speaking into the loudspeaker (convertAndSend). PaymentSubscriber is every classroom listening. That's the whole trick — Redis just relays the shout instantly to every listener.

// PaymentNotifier.java — the publishing side (any server can shout)
@Service
public class PaymentNotifier {
    private final StringRedisTemplate redis;

    public void notifyMoneyReceived(String toUser, String fromUser, int amount) {
        String message = """
            {"received":%d,"from":"%s"}""".formatted(amount, fromUser);
        redis.convertAndSend("payments:user:" + toUser, message);   // PUBLISH — fire and move on
    }
}
// PaymentSubscriber.java — the listening side (every server runs this)
@Configuration
public class PaymentSubscriber {

    @Bean
    public RedisMessageListenerContainer listenerContainer(RedisConnectionFactory factory,
                                                            PushService pushService) {
        RedisMessageListenerContainer container = new RedisMessageListenerContainer();
        container.setConnectionFactory(factory);

        container.addMessageListener((message, pattern) -> {
            String channel = new String(message.getChannel());        // payments:user:ramesh
            String user = channel.substring("payments:user:".length());
            String payload = new String(message.getBody());
            pushService.pushIfConnectedHere(user, payload);            // only the server holding his connection acts
        }, new PatternTopic("payments:user:*"));                       // SUBSCRIBE to all payment channels

        return container;
    }
}
What is happening underneath
  1. convertAndSend = PUBLISH. Redis looks at its list of current listeners for that channel and forwards the message to each one instantly — then forgets the message. Nothing is stored.
  2. The PatternTopic payments:user:* means each server hears ALL users' payment messages and simply ignores the ones for users not connected to it. Simple, and plenty fast.
  3. Ramesh's popup now appears in ~50 milliseconds — while Suresh is still holding his phone.
Best practices
  • Pub/Sub forgets. The transaction itself must already be safe in the database BEFORE you shout. Store first, shout second.
  • Use it for popups, live badges, "typing…" — never as the only record of anything.
  • One channel per user keeps the routing dead simple.

Redis Search — the Amazon search box

📖 Feature 3 from the concepts tab
1 · The problem

A shopper types "iph" and expects "iPhone 15", "iPhone charger", "iPhone case" before the next letter. That is one search per keystroke, per shopper, needing a ~2ms answer. Text-matching on the main database per keystroke would crush it.

2 · How Redis solves it

We store each product as a Redis Hash under product:..., and tell Redis Enterprise Cloud's search engine ONCE to index them. From then on, every product we save is automatically searchable, and each keystroke becomes one FT.SEARCH call answered from memory — bestsellers first.

📖 The story behind the code

Imagine a librarian who, every time a new book arrives, quietly adds an entry to the index at the back of a giant book — noting the title, author, and page. When a reader shouts "any book with 'iPhone' in it?", the librarian doesn't read every book — she flips to the index, finds "iPhone", and points to the right shelves in seconds.

SearchIndexInitializer is us telling the librarian "watch the products shelf" — once, at app startup. From then on, every product we save (through saveProduct) automatically gets an index entry — we don't do anything extra. ProductSearchService.suggest is the reader shouting a prefix; Redis returns matching products, bestsellers first, in a couple of milliseconds.

// SearchIndexInitializer.java — create the index once at startup (safe to re-run)
@Component
public class SearchIndexInitializer {
    private final StringRedisTemplate redis;

    @PostConstruct
    public void createIndex() {
        try {
            redis.execute((RedisCallback<Object>) conn -> conn.execute("FT.CREATE",
                "idx:products".getBytes(), "ON".getBytes(), "HASH".getBytes(),
                "PREFIX".getBytes(), "1".getBytes(), "product:".getBytes(),
                "SCHEMA".getBytes(),
                "name".getBytes(), "TEXT".getBytes(), "SORTABLE".getBytes(),
                "price".getBytes(), "NUMERIC".getBytes(), "SORTABLE".getBytes(),
                "sales".getBytes(), "NUMERIC".getBytes(), "SORTABLE".getBytes()));
        } catch (Exception alreadyExists) { /* index exists — that's fine */ }
    }
}
// ProductSearchService.java
@Service
public class ProductSearchService {
    private final StringRedisTemplate redis;

    // Save/update a product — the index picks it up automatically because of the "product:" prefix
    public void saveProduct(String id, String name, int price, int sales) {
        redis.opsForHash().putAll("product:" + id,
            Map.of("name", name, "price", String.valueOf(price), "sales", String.valueOf(sales)));
    }

    // Called on every keystroke: FT.SEARCH idx:products "iph*" SORTBY sales DESC LIMIT 0 5
    public List<ProductSuggestion> suggest(String typed, int max) {
        Object raw = redis.execute((RedisCallback<Object>) conn -> conn.execute("FT.SEARCH",
            "idx:products".getBytes(),
            (typed + "*").getBytes(),
            "SORTBY".getBytes(), "sales".getBytes(), "DESC".getBytes(),
            "LIMIT".getBytes(), "0".getBytes(), String.valueOf(max).getBytes()));
        return SearchResultParser.toSuggestions(raw);
    }
}
// SearchController.java — the REST API
@RestController
@RequestMapping("/api/v1/search")
public class SearchController {
    private final ProductSearchService searchService;

    @GetMapping("/products")                                  // GET /api/v1/search/products?q=iph
    public List<ProductSuggestion> products(@RequestParam String q,
                                             @RequestParam(defaultValue = "5") int max) {
        return searchService.suggest(q, max);
    }
}
What is happening underneath
  1. The index watches the prefix product:. Every hash we write under that prefix is indexed automatically — there is no "sync the index" job to build or forget.
  2. FT.SEARCH "iph*" uses the back-of-the-book index to jump straight to matching products — it never scans your data. ~2ms even with millions of products.
  3. SORTBY sales DESC puts bestsellers first, which is what shoppers actually want to see.
Best practices
  • Debounce on the client: search after a ~150ms typing pause, not literally every key press.
  • The index is a rebuildable copy — the main database remains the source of truth for products.
  • LIMIT your results server-side (5–8 suggestions) — nobody reads suggestion #40.

Redis Geospatial — “your delivery partner is 1.2 km away”

📖 Feature 4 from the concepts tab
1 · The problem

Thousands of delivery partners move around the city, sending their position every few seconds. Every customer opening the tracking screen asks: "which partners are near MY address, right now?" Measuring the distance to every partner, per customer, per refresh, is far too slow.

2 · How Redis solves it

Partners' apps update their position with one command (which overwrites in place — perfect for moving targets). The tracking screen asks "who is within 3 km of this point?" with one command — answered in about a millisecond, nearest first, distances included.

📖 The story behind the code

Picture a giant map of Pune pinned to a wall. Every delivery partner has a little magnetic pin with their name on it. Their app tells us "I'm at this corner now" — we walk over and move their pin to that spot. When a customer opens the tracking screen, we simply put a finger on the customer's address and read out the names of the nearest pins — Raju is 1.2 km, Salim is 2.4 km.

updateLocation moves the pin (GEOADD). findNearby reads the nearest pins to a point (GEOSEARCH). Redis does the "which pins are near this finger?" math instantly — no distance-calculation loop in your Java code.

// DeliveryLocationService.java
@Service
public class DeliveryLocationService {
    private final StringRedisTemplate redis;
    private static final String KEY = "partners:pune";    // one key per city

    // Called by the partner's app every few seconds
    public void updateLocation(String partnerId, double longitude, double latitude) {
        redis.opsForGeo().add(KEY, new Point(longitude, latitude), partnerId);   // GEOADD — overwrite in place
    }

    // Called by the customer's tracking screen
    public List<NearbyPartner> findNearby(double longitude, double latitude, double radiusKm, int max) {
        GeoResults<RedisGeoCommands.GeoLocation<String>> results = redis.opsForGeo().search(
            KEY,
            GeoReference.fromCoordinate(longitude, latitude),      // centered on the customer's address
            new Distance(radiusKm, Metrics.KILOMETERS),
            RedisGeoCommands.GeoSearchCommandArgs.newGeoSearchArgs()
                .includeDistance().sortAscending().limit(max));     // nearest first

        return results.getContent().stream()
            .map(r -> new NearbyPartner(r.getContent().getName(),
                                         r.getDistance().getValue()))   // km away
            .toList();
    }
}
// DeliveryController.java — the REST APIs
@RestController
@RequestMapping("/api/v1/delivery")
public class DeliveryController {
    private final DeliveryLocationService locationService;

    @PutMapping("/partners/{partnerId}/location")              // partner app calls this every few seconds
    public ResponseEntity<Void> updateLocation(@PathVariable String partnerId,
                                                @RequestBody LocationRequest request) {
        locationService.updateLocation(partnerId, request.longitude(), request.latitude());
        return ResponseEntity.noContent().build();
    }

    @GetMapping("/partners/nearby")     // GET /api/v1/delivery/partners/nearby?lon=73.856&lat=18.520&radiusKm=3
    public List<NearbyPartner> nearby(@RequestParam double lon, @RequestParam double lat,
                                       @RequestParam(defaultValue = "3") double radiusKm) {
        return locationService.findNearby(lon, lat, radiusKm, 10);
    }
}
What is happening underneath
  1. GEOADD turns lat+lon into one clever number where nearby places get nearby numbers, and stores partners sorted by it. "Near me" becomes "look at my number's neighbors" — a lookup, not a city-wide calculation.
  2. Updates are just GEOADD again — the partner's number changes, their position in the sorted structure moves. No delete-and-re-add.
  3. includeDistance + sortAscending gives the screen exactly what it displays: "Raju — 1.2 km", nearest first.
Best practices
  • One geo key per city/zone keeps searches small and fast.
  • Sweep out partners whose position hasn't updated recently (they went offline) — stale dots on a map break trust.
  • Location is personal data: collect it only with consent and only while needed.

Data Structures — five small features, five shapes

📖 Feature 5 from the concepts tab — all five, with code
📖 The story behind this whole section

We're going to build five tiny features — one for each data shape Redis offers. Each one is a real problem a real app faces every day, solved in about 15 lines of Java. Read the story, then the code — you'll see the shape click into place naturally.

String → YouTube-style view counter

1 · The problem

8,000 views per second on a viral video. "Read the count, add 1, write it back" from 20 servers loses views when two servers read the same number at the same moment.

2 · The fix

increment() — Redis adds 1 as one unbreakable step. No reads, no gaps, no lost views.

@Service
public class ViewCounterService {
    private final StringRedisTemplate redis;

    public long addView(String videoId) {
        return redis.opsForValue().increment("video:" + videoId + ":views");   // INCR — atomic +1
    }
    public long getViews(String videoId) {
        String v = redis.opsForValue().get("video:" + videoId + ":views");
        return v == null ? 0 : Long.parseLong(v);
    }
}

// REST: POST /api/v1/videos/{id}/view   → returns the new count

Hash → payment-app wallet

1 · The problem

A wallet has balance, currency, status. As one JSON blob, changing just the balance means rewriting everything — wasteful and racy.

2 · The fix

A Hash = one key with labeled compartments. Touch one field; leave the rest alone. HINCRBY changes the balance atomically.

@Service
public class WalletService {
    private final StringRedisTemplate redis;

    public void deduct(String user, int amount) {
        redis.opsForHash().increment("wallet:" + user, "balance", -amount);   // HINCRBY — one field, atomic
    }
    public String getBalance(String user) {
        Object b = redis.opsForHash().get("wallet:" + user, "balance");       // HGET — one field
        return b == null ? "0" : b.toString();
    }
}

// REST: GET /api/v1/wallet/{user}/balance

List → Facebook-style notifications

1 · The problem

Notifications: newest on top, 20 per page, scroll for more. Rebuilding this list from the database on every open is repeated, pointless work.

2 · The fix

A List: push new items to the front, read pages by range, trim the tail so it never grows forever.

@Service
public class NotificationService {
    private final StringRedisTemplate redis;

    public void add(String user, String notificationJson) {
        String key = "notifications:" + user;
        redis.opsForList().leftPush(key, notificationJson);   // LPUSH — newest at the front
        redis.opsForList().trim(key, 0, 499);                  // LTRIM — keep newest 500
    }
    public List<String> page(String user, int page, int size) {
        long start = (long) page * size;
        return redis.opsForList().range("notifications:" + user, start, start + size - 1);  // LRANGE
    }
}

// REST: GET /api/v1/notifications?user=ramesh&page=0

Set → Instagram likes

1 · The problem

Fast double-taps on a laggy network count one person as two likes. And the app must know instantly whether YOUR heart should be red on each post.

2 · The fix

A Set holds unique members — duplicates are impossible by definition. The add command itself answers "new like" (1) or "already liked" (0); the count is the set's size; the red heart is one membership check.

@Service
public class LikeService {
    private final StringRedisTemplate redis;

    public boolean like(String postId, String user) {
        Long added = redis.opsForSet().add("post:" + postId + ":likers", user);   // SADD
        return added != null && added == 1;      // double-tap-proof: second tap returns 0
    }
    public long count(String postId) {
        Long c = redis.opsForSet().size("post:" + postId + ":likers");            // SCARD
        return c == null ? 0 : c;
    }
    public boolean hasLiked(String postId, String user) {                            // SISMEMBER
        return Boolean.TRUE.equals(redis.opsForSet().isMember("post:" + postId + ":likers", user));
    }
}

// REST: POST /api/v1/posts/{id}/like?user=ramesh   |   GET /api/v1/posts/{id}/likes/count

Sorted Set → stock-trading “Top Gainers”

1 · The problem

Top Gainers ranked by today's % change, updating every second, watched by every user. Re-sorting thousands of stocks per second in a database is hopeless.

2 · The fix

A Sorted Set keeps members ordered by score AT ALL TIMES. Update a stock's % change and it slides to its new position instantly. "Top 10" is a read, never a sort.

@Service
public class GainersService {
    private final StringRedisTemplate redis;
    private static final String KEY = "gainers:today";

    public void updateChange(String symbol, double percentChange) {
        redis.opsForZSet().add(KEY, symbol, percentChange);        // ZADD — score updated, order self-adjusts
    }

    public List<Gainer> top(int n) {
        Set<ZSetOperations.TypedTuple<String>> top =
            redis.opsForZSet().reverseRangeWithScores(KEY, 0, n - 1);   // ZREVRANGE — top N, already sorted
        List<Gainer> out = new ArrayList<>();
        int rank = 1;
        for (var t : top) out.add(new Gainer(rank++, t.getValue(), t.getScore()));
        return out;
    }

    public Long rankOf(String symbol) {
        Long r = redis.opsForZSet().reverseRank(KEY, symbol);       // ZREVRANK — instant rank lookup
        return r == null ? null : r + 1;
    }
}

// REST: GET /api/v1/stocks/gainers/top   |   GET /api/v1/stocks/{symbol}/rank
What is happening underneath (all five)
  1. Every command above is atomic — Redis's one-at-a-time cashier means INCR, HINCRBY, SADD, and ZADD can never half-happen or collide. All the race conditions you'd have to code around elsewhere simply don't exist here.
  2. Each shape answers its question in one step: the count IS the set's size; the top 10 IS the first 10; the page IS the range. No loops, no sorting, no assembly in Java.
Best practices
  • Match the shape to the question: unique? → Set. Ranked? → Sorted Set. Newest-first pages? → List. One field of an object? → Hash. A single value or counter? → String.
  • Cap everything that grows: LTRIM lists, expire finished days' sorted sets, archive old data to the database.
  • Name keys like folders (post:9001:likers) — six months later, anyone can read your Redis like a map.

Atomic Counter — the Flash Sale that sells exactly 100

📖 Feature 6 from the concepts tab — the concurrency showpiece
1 · The problem

Flipkart is running a flash sale: 100 phones, ₹9,999 each, live at 12:00 sharp. At exactly 12:00:00, our servers receive 50,000 "BUY" clicks in a single second. Our backend has 50 servers behind a load balancer; each server handles many requests in parallel using Java virtual threads. The rule is simple and unforgiving: sell exactly 100. Not 99. Not 101. Exactly 100.

Our first version used AtomicInteger inside each server to count down. Result: we sold 4,700 phones we didn't have. Why? Because AtomicInteger is only atomic INSIDE one JVM. All 50 servers had their own separate counter — each one happily sold "the last 100 phones."

2 · How Redis solves it

We move the counter OUT of every server's memory and INTO Redis. Now all 50 servers share ONE counter. Each buy click sends one DECR command; Redis executes them one at a time (single cashier — remember Section 4), so no two DECRs can ever collide. The command returns the new value — so we know immediately whether the sale succeeded or the shelves are empty. If the counter went negative (we were too late), we INCR it back and reject the sale. Simple, exact, correct at any traffic level.

📖 The story behind the code

Think of the flash sale as a big glass jar with exactly 100 marbles. The jar sits in one place (Redis). All 50 servers reach in through a small opening — but the opening is narrow, so only one hand fits at a time. Each customer says "give me a marble," a hand grabs one, the jar's counter goes down by 1, and the customer walks away with the marble. When the jar is empty, the next hand comes back with nothing — Redis returns -1, and we tell the customer "sold out, better luck next time."

Look for these pieces in the code below:

  • startSale — puts the 100 marbles in the jar.
  • attemptPurchase — one hand reaches in (DECR). If it got a marble (result ≥ 0), the sale succeeds. If it got -1 (empty), we put the imaginary marble back (INCR) and reject the sale.
  • ExecutorService with virtual threads — this is Java 25's superpower: we can launch 50,000 concurrent buy attempts in one JVM cheaply. It also proves the point: even with that much concurrency slamming Redis, the count comes out exactly right.
// FlashSaleService.java — the whole flash sale is ~15 lines
@Service
public class FlashSaleService {

    private final StringRedisTemplate redis;
    private static final String STOCK_KEY = "stock:flashsale:iphone15";

    public FlashSaleService(StringRedisTemplate redis) {
        this.redis = redis;
    }

    // Called once by the admin before the sale opens
    public void startSale(int units) {
        redis.opsForValue().set(STOCK_KEY, String.valueOf(units));
        redis.expire(STOCK_KEY, Duration.ofHours(2));       // self-cleanup after 2 hours
    }

    // Called by every "BUY" click. Returns a record with the outcome.
    public PurchaseResult attemptPurchase(String userId) {
        Long remaining = redis.opsForValue().decrement(STOCK_KEY);   // DECR — atomic across ALL 50 servers

        if (remaining == null) {                    // sale hasn't started or already expired
            return new PurchaseResult(false, "Sale not active", -1);
        }
        if (remaining < 0) {                        // too late — someone else got the last one
            redis.opsForValue().increment(STOCK_KEY);   // put the "imaginary" marble back
            return new PurchaseResult(false, "Sold out", 0);
        }
        return new PurchaseResult(true, "Reserved for " + userId, remaining.intValue());
    }

    public int remaining() {
        String v = redis.opsForValue().get(STOCK_KEY);
        return v == null ? 0 : Math.max(0, Integer.parseInt(v));
    }
}

// Java 25 record — the neat return type for the outcome
public record PurchaseResult(boolean success, String message, int remaining) {}
// FlashSaleController.java — the REST API
@RestController
@RequestMapping("/api/v1/flashsale")
public class FlashSaleController {

    private final FlashSaleService flashSaleService;

    public FlashSaleController(FlashSaleService flashSaleService) {
        this.flashSaleService = flashSaleService;
    }

    @PostMapping("/start")                          // admin: POST /api/v1/flashsale/start?units=100
    public ResponseEntity<String> start(@RequestParam int units) {
        flashSaleService.startSale(units);
        return ResponseEntity.ok("Sale started with " + units + " units");
    }

    @PostMapping("/buy")                            // customer: POST /api/v1/flashsale/buy?userId=ramesh
    public ResponseEntity<PurchaseResult> buy(@RequestParam String userId) {
        PurchaseResult result = flashSaleService.attemptPurchase(userId);
        return result.success()
            ? ResponseEntity.ok(result)
            : ResponseEntity.status(HttpStatus.CONFLICT).body(result);   // 409 = "sold out"
    }

    @GetMapping("/remaining")                       // GET /api/v1/flashsale/remaining
    public int remaining() {
        return flashSaleService.remaining();
    }
}
// FlashSaleConcurrencyTest.java — proof it works under 50,000 concurrent clicks
// This uses Java 25's virtual threads to simulate 50,000 people clicking at the same instant.
@SpringBootTest
class FlashSaleConcurrencyTest {

    @Autowired FlashSaleService flashSaleService;

    @Test
    void sellsExactly100PhonesEvenWith50000ConcurrentClicks() throws Exception {
        flashSaleService.startSale(100);
        AtomicInteger successes = new AtomicInteger();
        AtomicInteger failures  = new AtomicInteger();

        // Java 25 virtual threads — 50,000 of them, cheap to create
        try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
            List<Future<?>> futures = new ArrayList<>();
            for (int i = 0; i < 50_000; i++) {
                final int userNum = i;
                futures.add(executor.submit(() -> {
                    var result = flashSaleService.attemptPurchase("user" + userNum);
                    (result.success() ? successes : failures).incrementAndGet();
                }));
            }
            for (Future<?> f : futures) f.get();       // wait for everyone
        }

        assertEquals(100, successes.get());              // exactly 100 phones sold ✓
        assertEquals(49_900, failures.get());            // exactly 49,900 got "sold out" ✓
        assertEquals(0, flashSaleService.remaining());   // counter safely at 0 ✓
    }
}
🔍 What is happening underneath
  1. DECR is one atomic step in Redis's single cashier queue. Even if 50,000 DECR commands arrive in the same millisecond, they line up and execute one after the other — no two can read the same "current value." Every DECR gets a unique return value.
  2. The return value is the whole decision: ≥ 0 means "you got a phone." −1, −2, −3, ... mean "you were the 101st, 102nd, 103rd person — sold out." We compensate the negative values with INCR so the counter settles at exactly 0 when the dust clears.
  3. Java 25 virtual threads let the test simulate real load — 50,000 concurrent HTTP-like calls in one JVM without breaking a sweat. If AtomicInteger was used instead, the test would still pass on ONE server — and quietly explode when deployed to 50 servers, because each JVM has its own AtomicInteger. Redis is one shared truth for all of them.
Best practices
  • For anything shared across servers — flash-sale stock, ticket seats, API quotas, rate limits — use Redis atomic commands. Never rely on AtomicInteger or synchronized; those only work inside one JVM.
  • When over-decrementing is possible, always check the return value and compensate with INCR — it's one extra command and it keeps the counter correct.
  • Give short-lived counters (flash sales, rate-limit windows) a TTL so they clean themselves up.
  • For counters where the money is on the line (real payments, real inventory), also write a durable record to the main database on success — the Redis counter is the fast referee, the database is the ledger.
